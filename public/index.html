PostgreSQL GUI Guide

Everything you need to know to manage your database effectively.

1. Browsing Tables

The Explorer sidebar on the left displays all public tables in your database. Click on any table to instantly view its schema and first 100 rows. The sidebar highlights the active table with a premium indicator, and automatically populates the SQL Editor with a simple SELECT query.

SELECT * FROM users LIMIT 100;

2. Writing & Executing Queries

The Query Editor is fully featured with syntax highlighting and smart theming. You can run queries by clicking the Run Query button or by pressing Cmd/Ctrl + Enter. The editor scales gracefully and results are presented in a responsive data grid.

Errors are handled elegantly via toast notifications and detailed inline error states in the data grid, allowing you to debug SQL syntax issues instantly without losing context.

3. Safe Mode & Reverting Data

When the Safe Mode checkbox is enabled, the GUI creates a microscopic snapshot of your database milliseconds before executing your query. If you accidentally execute a destructive query (like a DELETE without a WHERE clause), a Revert button will appear in the toolbar.

Note: Safe Mode uses `pg_dump` internally. For massive databases, Safe Mode might add a slight delay to query execution.

4. Backups & Restoration

Use the icons in the far-left Activity Bar to trigger backups. You can backup the entire database, or use the "Backup" button above the Editor to backup just the specific table you are currently viewing. You can choose between Plain Text (.sql) and Custom Binary (.dump) formats.

To restore data, click the icon. It accepts raw .sql and .dump files via file selection or intuitive drag-and-drop, providing immediate visual feedback upon attachment. Be careful—restoring data may overwrite existing tables depending on the backup's structure.

5. Limitations

  • Foreign Keys & Reverting: The "Safe Mode Revert" feature drops the table before restoring it to prevent duplicate key errors. However, if the table is referenced by a Foreign Key from another table, PostgreSQL will reject the DROP command to protect the relationship, causing the revert to fail.
  • Large Databases: Backups and restores run synchronously within the HTTP request. If you attempt to dump/restore massive databases (e.g. multi-gigabyte), your browser or Node.js may time out the connection. For large databases, it is recommended to use the .dump custom binary format, which is compressed.
  • Query Limits: Unbounded queries (e.g. SELECT * FROM giant_table) will pull all rows into Node's memory. It is expected that developers write their own LIMIT clauses to avoid crashing the server.
  • Local Focus: The server binds to 127.0.0.1 by default. To expose it over a network, set the HOST environment variable explicitly.

6. Read-Only Access

If configured by the administrator, logging in with a Read-Only password will restrict the interface to safe operations. You will only be able to execute SELECT queries. This is strictly enforced at the database transaction layer, ensuring destructive queries (like INSERT, UPDATE, DELETE, DROP) are impossible. Backup/Restore functionality will also be disabled.

7. Security & Rate Limiting

This GUI employs robust security measures including built-in CSRF protection. Login endpoints are rate-limited to 5 attempts per 15 minutes to prevent brute-force attacks, while query and backup operations are also limited to prevent resource exhaustion. Passwords are cryptographically hashed using `bcrypt`, and the application leverages `helmet` to set strict HTTP security headers.
